# `db:seed-demo` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Construir `bun db:seed-demo` — um comando que reconstrói a DB de dev com um fixture que honra todos os invariantes de domínio.

**Architecture:** Um orquestrador (`scripts/seed-demo.ts`) abre uma transação Drizzle e chama módulos por domínio (`scripts/seed/*.ts`) em ordem de dependência; cada módulo insere seu slice e propaga IDs via um objeto `SeedContext`. Um módulo `verify.ts` roda no fim e dá rollback se algum invariante falhar.

**Tech Stack:** Bun, Drizzle ORM 0.45 + node-postgres, TypeScript. IDs via `crypto.randomUUID()`.

**Spec:** `docs/superpowers/specs/2026-05-17-seed-demo-design.md` (leia antes de começar).

**Branch:** `feat/seed-demo` (já em checkout).

**Pré-requisito de runtime:** `bun db:sync` aplicado (schema + triggers de `category` path/depth) antes de rodar o seed.

```ts
// packages/db/scripts/seed/context.ts
import type { ExtractTablesWithRelations } from "drizzle-orm";
import type { NodePgQueryResultHKT } from "drizzle-orm/node-postgres";
import type { PgTransaction } from "drizzle-orm/pg-core";

// Tipo da transação Drizzle (o que `db.transaction(async (tx) => …)` entrega).
export type Tx = PgTransaction<
	NodePgQueryResultHKT,
	Record<string, never>,
	ExtractTablesWithRelations<Record<string, never>>
>;

export interface SeedContext {
	staffUserIds: string[]; // staff existentes — lidos, nunca criados
	branchIds: string[];
	defaultBranchId: string;
	supplierIds: string[];
	categoryIdBySlug: Record<string, string>;
	attributeIdBySlug: Record<string, string>;
	toolIds: string[];
	variantIdsByTool: Record<string, string[]>;
	defaultVariantByTool: Record<string, string>;
	primaryCategoryByTool: Record<string, string>;
	clientIds: string[];
	orderIds: string[];
}

export function emptyContext(staffUserIds: string[]): SeedContext {
	return {
		staffUserIds,
		branchIds: [],
		defaultBranchId: "",
		supplierIds: [],
		categoryIdBySlug: {},
		attributeIdBySlug: {},
		toolIds: [],
		variantIdsByTool: {},
		defaultVariantByTool: {},
		primaryCategoryByTool: {},
		clientIds: [],
		orderIds: [],
	};
}
```

## Task 2: `core.ts` — branches e user_branch

**Files:**
- Create: `packages/db/scripts/seed/core.ts`
- Modify: `packages/db/scripts/seed-demo.ts`

- [ ] **Step 1: Criar `core.ts`**

`seedCore` insere 3 branches (uma `isDefault`) e liga cada staff de `ctx.staffUserIds` às 3 branches via `user_branch`. Importar as tabelas de `@emach/db/schema/inventory`.

```ts
// packages/db/scripts/seed/core.ts
import { branch, userBranch } from "@emach/db/schema/inventory";
import type { SeedContext, Tx } from "./context";

const BRANCHES = [
	{ name: "Matriz — São Paulo", isDefault: true },
	{ name: "Filial — Campinas", isDefault: false },
	{ name: "Filial — Ribeirão Preto", isDefault: false },
];

export async function seedCore(tx: Tx, ctx: SeedContext): Promise<void> {
	for (const b of BRANCHES) {
		const id = crypto.randomUUID();
		await tx.insert(branch).values({ id, name: b.name, isDefault: b.isDefault });
		ctx.branchIds.push(id);
		if (b.isDefault) ctx.defaultBranchId = id;
	}
	for (const userId of ctx.staffUserIds) {
		for (const branchId of ctx.branchIds) {
			await tx.insert(userBranch).values({ userId, branchId });
		}
	}
}
```

> Conferir os campos reais de `branch` e `user_branch` em `packages/db/src/schema/inventory.ts` antes de escrever — usar exatamente as colunas do schema (ex.: se `branch` tiver `slug`, `city`, etc., preencher).

## Task 4: `inventory.ts` — stock_level + movimentos de abertura

**Files:**
- Create: `packages/db/scripts/seed/inventory.ts`
- Modify: `packages/db/scripts/seed-demo.ts`

- [ ] **Step 1: Criar `inventory.ts`**

Para cada variante (de `ctx.variantIdsByTool`) em 1–3 branches de `ctx.branchIds`:
- Definir uma quantidade de abertura `qty` (ex.: 10–80, alguns 0 para exercitar reorder).
- Inserir 1 `stock_movement` `reason='entrada_compra'`, `delta = qty`, `actorType='user'`, `actorId` = um `ctx.staffUserIds[0]`, `variantId`, `branchId`. (Conferir colunas em `packages/db/src/schema/stock-movements.ts`.)
- Inserir o `stock_level` `(variantId, branchId)` com `quantity = qty`, `minQty` e `reorderPoint` plausíveis (`reorderPoint >= minQty`, ambos ≥ 0).

A coerência (Task 8 verifica): `stock_level.quantity` == soma dos `stock_movement.delta` daquela `(variantId, branchId)`. Aqui só há a abertura; os `saida_venda` da Task 6 ajustam o saldo lá.

> **Importante:** como a Task 6 vai debitar estoque por vendas, decidir a `qty` de abertura **alta o suficiente** para cobrir o que os orders pagos vão consumir, senão o CHECK `quantity >= 0` falha. Alternativa mais simples: a Task 6, ao criar um `saida_venda`, **incrementa a abertura correspondente** — ver Task 6, Step 2.

## Task 6: `sales.ts` — orders, itens, histórico, débito de estoque, reviews

**Files:**
- Create: `packages/db/scripts/seed/sales.ts`
- Modify: `packages/db/scripts/seed-demo.ts`

- [ ] **Step 1: Orders + itens + histórico**

~17 `order`, com **pelo menos um por valor de `order_status`** (`pending_payment`, `payment_failed`, `paid`, `preparing`, `shipped`, `delivered`, `returned`, `refunded`, `canceled`). Cada order:
- `clientId` de `ctx.clientIds`, `number` único legível (ex.: `EM-2026-0001`), snapshots de valores e endereço, `branchId` de fulfillment.
- 1–4 `order_item`: `toolId` + `variantId` (de `ctx.variantIdsByTool`), `quantity`, snapshots fiscais/dimensão.
- `order_status_history`: trilha **completa** do `pending_payment` até o status atual, seguindo o caminho legal do ADR-0005:
  - `pending_payment → paid → preparing → shipped → delivered → returned → refunded`
  - `canceled` só a partir de `pending_payment`/`payment_failed`; `payment_failed` a partir de `pending_payment`.
  - Cada transição = uma linha com `fromStatus`/`toStatus` (conferir colunas reais em `packages/db/src/schema/orders.ts`), `actorType` (`system` para a ida até `paid`, `user` daí em diante), `actorUserId` coerente.

Guardar `ctx.orderIds`.

- [ ] **Step 2: Débito de estoque (`saida_venda`)**

Para cada `order_item` de order com status `paid` ou posterior (i.e. estoque já saiu): inserir um `stock_movement` `reason='saida_venda'`, `delta = -quantity`, `variantId`, `branchId` = a filial de fulfillment do order, `actorType='system'`, `actorId=null`. Respeitar o partial unique index de idempotência (a chave inclui a referência ao order/item — conferir em `stock-movements.ts`).
**Manter o saldo coerente e não-negativo:** ao debitar, somar de volta o mesmo total na quantidade de abertura — ou seja, em vez de mexer no `entrada_compra` já inserido, **incrementar o `entrada_compra` da Task 4** não é possível retroativamente; então a abordagem: a Task 4 grava `entrada_compra` com `qty` base, e aqui, para cada `saida_venda`, fazer `UPDATE stock_level SET quantity = quantity + delta WHERE variant_id/branch_id` (delta negativo) — e a Task 4 já garantiu `qty` base alto o suficiente (≥ soma das vendas daquela variante×filial). Para garantir isso sem ordem-de-dependência frágil: calcular as vendas **antes** e passar um mapa de "qty mínima por (variant,branch)" para `seedInventory`. Implementação concreta: `sales.ts` exporta `computeSalesDebits(ctx)` (puro, decide orders/itens) chamado por `seed-demo.ts` **antes** de `seedInventory`, e o resultado vai no `ctx`; `seedInventory` usa-o para dimensionar a abertura; `seedSales` então insere orders + `saida_venda` + faz o `UPDATE` do saldo.

> Reordenar o orquestrador para: `core → catalog → (computeSalesPlan) → inventory → clients → sales → marketing`. O "plano de vendas" é dados puros (quais orders, itens, quantidades); a inserção acontece em `seedSales`.

## Task 8: `verify.ts` — asserções de invariante

**Files:**
- Create: `packages/db/scripts/seed/verify.ts`
- Modify: `packages/db/scripts/seed-demo.ts`

- [ ] **Step 1: Criar `verify.ts`**

`verifySeed(tx)` roda uma lista de checagens; cada uma é uma query que **deve** retornar 0; se alguma retornar >0, lançar `Error` com o nome da regra violada. Checagens (uma query cada):

1. Toda tool tem default variant: `tool` sem `tool_variant is_default`.
2. Toda tool tem primary category: `tool` sem `tool_category is_primary`.
3. `tool_attribute_value` órfão de path: value cuja `attribute_definition.category_id` não está no `path` da primary category da tool.
4. Estoque reconcilia: `stock_level.quantity <> SUM(stock_movement.delta)` por `(variant_id, branch_id)`.
5. `stock_level.quantity < 0`.
6. Todo order tem `order_status_history`.
7. Todo order tem ≥1 linha de history com `to_status='pending_payment'` (a trilha começa na origem).
8. `review` com order não-`delivered`+ ou cuja tool não está no order.
9. Todo client tem `consent_log` de `tos` e de `privacy`.
10. Actor coherence: linhas `actor_type='user'` com actor nulo, ou `system` com actor não-nulo, em `stock_movement`/`order_status_history`/`client_audit_log`.

```ts
// estrutura
import { sql } from "drizzle-orm";
import type { Tx } from "./context";

const CHECKS: { name: string; query: string }[] = [
	{ name: "tool sem default variant", query: `SELECT count(*) AS n FROM tool t WHERE NOT EXISTS (SELECT 1 FROM tool_variant v WHERE v.tool_id=t.id AND v.is_default)` },
	// … as 10 checagens
];

export async function verifySeed(tx: Tx): Promise<void> {
	for (const c of CHECKS) {
		const res = await tx.execute<{ n: number }>(sql.raw(c.query));
		const n = Number(res.rows[0]?.n ?? 0);
		if (n > 0) throw new Error(`Invariante violado [${c.name}]: ${n} linha(s)`);
	}
}
```
