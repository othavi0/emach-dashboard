     Plano: Unificar category → product_type (Tool)

     Contexto

     O schema tool tem dois conceitos redundantes representando a mesma ideia de
      classificação:

     - category — tabela lookup M:1, user-editável via /dashboard/categories
     (ex: "Furadeiras", "Serras"). FK tool.categoryId é obrigatório no Zod mas
     nullable no DB.
     - productType — coluna text com enum fixo machine|equipment|part|accessory,
      nullable, sem FK. Mapeado para labels PT-BR (PRODUCT_TYPE_LABELS).

     No form de criação/edição (tool-form.tsx) os dois aparecem como selects
     separados; nos filtros (tool-filters.tsx) como dois params independentes
     (?category= e ?productType=). Isso confunde usuário interno e duplica
     decisão sem benefício — não há semântica distinta em uso.

     Objetivo: consolidar em um único conceito — tabela lookup flexível chamada
     product_type — eliminando o enum fixo e unificando UI, queries e tipos TS.
     Migração dev-only (monorepo ainda pré-prod), com script de backfill que
     preserva categorias existentes e garante que nenhuma tool fique órfã.

     Script de migração

     Caminho: scripts/migrate-category-to-product-type.ts (na raiz do monorepo
     ou em packages/db/scripts/).

     Fluxo em 3 fases (aditivo → backfill → cleanup):

     Fase A — push aditivo:
     1. Editar schema/tools.ts: adicionar productType pgTable (tabela nova,
     coexiste com category) e adicionar coluna productTypeId: 
     text("product_type_id") em tool (nullable ainda, sem FK NOT NULL). Manter
     category, categoryId, coluna enum productType, index e check existentes
     intactos.
     2. bun db:push (sem --force) — cria tabela nova e coluna aditiva. Zero
     risco de perda.

     Fase B — script de backfill (scripts/migrate-category-to-product-type.ts,
     tudo em uma transação):
     1. Read state — SELECT all de category (id, name, slug, description) e
     SELECT all de tool (id, categoryId, productType antigo).
     2. Seed 4 rows — INSERT em product_type com IDs via crypto.randomUUID().
     Guardar map seedMap = { machine: uuid1, equipment: uuid2, part: uuid3, 
     accessory: uuid4 }:
       - { name: "Máquina", slug: "machine" }
       - { name: "Equipamento", slug: "equipment" }
       - { name: "Peça", slug: "part" }
       - { name: "Acessório", slug: "accessory" }
     3. Copia categorias existentes — para cada row em category, INSERT em
     product_type com mesmo id, name, description. Slug = category.slug se
     não-NULL, senão slugify(name) (função já existe em tool-schema.ts:104-112;
     reusar). Se houver colisão de slug com seed, sufixar -N.
     4. Backfill tool.product_type_id — lógica por tool, em UPDATE batch:
     se tool.productType IS NOT NULL (enum antigo):
       tool.product_type_id = seedMap[tool.productType]   // productType vence
     conflito
     senão se tool.categoryId IS NOT NULL:
       tool.product_type_id = tool.categoryId             // preserva vínculo
     com categoria migrada
     senão:
       tool.product_type_id = seedMap["machine"]          // default órfãs
     5. Verificação — SELECT COUNT(*) FROM tool WHERE product_type_id IS NULL
     deve retornar 0. Se > 0, ROLLBACK e abortar.
     6. Log resumo — "Migrados: N tools (K preservadas via categoryId, L
     relinkadas via productType, M default), X categorias existentes copiadas, 4
      seeds, 0 órfãs."

     Fase C — push cleanup:
     1. Editar schema/tools.ts novamente:
       - Tornar productTypeId NOT NULL + adicionar references(() => 
     productType.id, { onDelete: "restrict" }).
       - Remover pgTable category, relations categoryRelations.
       - Remover em tool: coluna productType (enum), index
     tool_product_type_idx, check valid_product_type, coluna categoryId, index
     tool_category_id_idx.
       - Atualizar toolRelations pra usar novo FK.
     2. bun db:push --force — dropa category, dropa 2 colunas antigas em tool,
     aplica NOT NULL + FK nova.

     Por que 3 fases: garante zero janela em que tool pode ficar órfã. Fase A é
     reversível (só adiciona). Fase B roda em transação (all-or-nothing). Fase C
      só acontece depois de verificação manual de que o backfill passou — só aí
     --force é seguro.
